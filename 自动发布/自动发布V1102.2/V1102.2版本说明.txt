三、业务规则(发文上限 / 补漏 / quota)
============================================================
【3.1 发文上限(平台规则)】
  ┌──────────┬──────────┬─────────────────────────────────┐
  │ 件       │ 单账号上限│ 补漏规则                        │
  ├──────────┼──────────┼─────────────────────────────────┤
  │ 微头条   │ 5 篇/天  │ 漏 1/2/3/4/5 各补对应数;已发 5 不动 │
  │ 文章     │ 3 篇/天  │ 漏 1/2/3 各补;已发 3 不动      │
  │ 文章定时 │ 3 篇/天  │ 同文章                          │
  └──────────┴──────────┴─────────────────────────────────┘

  说明:
   - 上限来自头条平台限制,超了平台限流
   - 补漏 = target − 实际成功篇数
   - 「永久跳过」账号不算在内(完全不发)

【3.2 [v1102] quota 动态算法】
  公式:
    quota_total = (已发总篇数 + 当前素材池数) // 账号数

  例 1: 750 篇池 / 150 账号,发 2 轮(300 已发) + 用户拿走 300 → 池剩 150
    quota_total = (300 + 150) // 150 = 3 轮
    每号缺 = 3 - 2 = 1 轮 = 还要发 150 篇

  例 2: 235 篇池 / 150 账号,已发 14 篇
    quota_total = (14 + 235) // 150 = 1 轮(取整)
    每号缺 = 1 - 已发本人 count

  封顶: quota 取整后封顶各件日上限(微头条 5 / 文章 3 / 文章定时 3)
  核心: 每号 quota 比"只看池剩"算法更准,能反映"用户拿走部分但要补"语义

【3.3 [v1102] 单大循环 / 小轮定义】
  1 次发文 = 1 大循环 = N 小轮(N = 素材数 // 账号数,动态算)
  1 小轮 = 全账号各发 1 篇
  例: 750 / 150 = 5 小轮 = 1 大循环

  Phase A(主流程):每小轮全账号轮一遍发 1 篇
  Phase B(补发):  小轮末从「失败列表」补发本轮失败的账号
  「本轮已发」 sheet 不在小轮末 clear,**累积到大循环末**(齐活才清)

【3.4 重扫素材池机制】
  主循环跑完一轮后:
    1. _resync_pool 重扫素材/ 目录:剔除幽灵引用 + 加载新素材
    2. doc_pool 实时更新 (v1101.4)
    3. 如果池剩 > 0 + 还有 acc_count<quota 账号 → 继续下一轮
    4. 如果都满 quota → "等 60s 重扫素材池" 死循环(等用户加新素材)
       ⚠️ 这死循环不会自动重算 quota,加新素材后 quota 不会上调
       (这是 v1102 已知设计 quirk,用户加素材后建议手动重启)

【4.1 业务语义(缺哥拍板 2026-05-04)】
  ┌─────────────────────────────────────────────────────────┐
  │ 中断在第 X 轮第 Y 账号 → 重启从该轮该账号继续:         │
  │   上面 (< Y):已发 X 轮 → sheet count=X → 跳过          │
  │   中断+下面 (≥ Y):已发 X-1 轮 → sheet count=X-1 →      │
  │     排第 X 小轮各 1 篇                                  │
  │ 上面已完成,下面没完成 → 自然以中断点为新起点继续       │
  │ "本轮不重发 + 缺 N 补 N" = 业务规则推出来的"从中断处起"│
  └─────────────────────────────────────────────────────────┘

【4.2 catchup.py 工具(v1101.5 沉淀)— 用户主动跑】
  诞生:2026-05-02 04:09 电信断网事件(全队 394 账号误判硬终止),

    2. 算漏数:优先「待补漏」 sheet,fallback 看 log 算 quota - 已成功

【4.3 [v1102] 主线内化中断恢复(2026-05-04 设计反转 v1101.5)】
  架构反转:v1101.5 「主线零内部控制 + catchup 独立」 →
            v1102 「主线 gtg_batch / gtg_timer 自己内化中断恢复」

  核心 7 点:
    1. 「本轮已发」 sheet 加「已发次数」列 (账号, count)
    2. sheet 不在小轮末 clear,累积到大循环末才 clear
    3. 每次成功发文 → sheet 该账号 count += 1
    4. 主线启动 → 读 sheet 重建 acc_count[账号] = count
    5. quota 动态算 = (已发总 + 池剩) // 账号数(见 3.2)
    6. 每号缺 N = quota - acc_count[账号](负数当 0,跳过)
    7. 主循环 acc_count < quota 自然停 = 不超不漏

  对应代码改动:
    - batch.py: _clear_round_sheets() 调用从小轮末移到大循环末
    - batch.py: _append_sent_excel(name) 改成 (账号, count) 累加
    - batch.py main(): 启动时 _read_sent_with_count → initial_acc_count
    - timer.py: 同款机制(timer 件之前没这套,要补全)
    - timer.py _expand_tasks: q=2 排 中+晚 / q=1 排晚

  实证(NEO2 文章定时件):
    117 已发后中断 → 注入 sheet (117 行 count=1) + 启 timer →
    log "本轮已发 117 篇,本轮还需排 333 篇"
    数学正确,quota 不超

【4.4 [v1102.1] hotfix — 主循环按白名单顺序重排】
  问题: catchup 写白名单是环形重排(断点置首),但主线只过滤不重排:
    accounts = [a for a in accounts if any(wn in a or a in wn for wn in wl_map)]
  → accounts 顺序还是原顺序,catchup 重排白名单白搭。

  修法:加载白名单后,按白名单顺序重排 accounts:
    _orig_accounts = list(accounts)
    _new_accounts = []
    _seen = set()
    for wn, _wq in whitelist_with_q:    # Win = whitelist_with_q,Mac = _wl_q
        for a in _orig_accounts:
            if a not in _seen and (wn in a or a in wn):
                _new_accounts.append(a)
                _seen.add(a)
                break
    accounts = _new_accounts

============================================================
【14.1 推稿铁律(2026-05-04 NEO 470 篇丢失事故强化)】
  核心: scp 是 cp,不是 mv → 推送 → ssh 远端确认数对 → 才 rm 源

  禁忌(违反 = 数据丢失风险):
    - ✗ scp 完成后假设源被删
    - ✗ 用 pipe 命令 exit code 判断成功(`tar | ssh "tar -xzf"` 即使
        包是空的也 exit 0)
    - ✗ tar --include 跨 GNU/BSD 不一致(可能生成空包)

  正解(每次推稿必走):
    1. 源池 mv 到 stage(中转目录)
    2. scp/tar 推到目标机
    3. ssh 远端 ls *.docx | wc -l 实查 = 推送数
    4. **数字对账匹配** 才 rm 源(不准用 wildcard,显式列文件)
    5. 检查清单: find <源> -maxdepth 1 -name '*.docx' | wc -l = 0

  推稿模板:
    tar -czf - -C "$SRC" . | ssh kenchoiXXX@100.x \
      "cd $TGT && tar -xzf - && ls *.docx | wc -l"
    # ↑ 看输出 == 预期推送数,才 rm 源
    cd "$SRC" && rm 显式文件1.docx 显式文件2.docx ...

【14.2 跨机标题查重防类同(防风控)】
  风险: 4 Mac 同期发文,标题相似 → 平台扣分(类同认定)
  防范:
    1. 推稿前用 sed 提取标题主体(去 NNN 编号 + 时间戳):
       sed -E 's/^[0-9]+_[0-9.]+_//; s/_[0-9]{14}\.docx$//'
    2. 跨机 comm -12 看是否有同标题
    3. 有同标题 = 跨机内容可能重复,触发风控
    4. 类同扣分 ≈ 50 信用分/篇,严重

【14.3 推前确认目标路径(2026-04-30 实证)】
  正确目标: ~/Desktop/<件名>/素材/(件子目录)
  ✗ 错误别名:~/Desktop/自动微头条素材/(顶级,代码不读)

  推前必做:
    grep -nE "DOCS_FOLDER|SOURCE|BASE_DIR" 大件/gtg_*.py
  确认代码读的真实路径,别凭直觉选目录

  现象: timer 23:49 启动,跑到次日凌晨,RUN_REPORT_DIR 仍是前日
  应对:
    1. 跨天后所有日志都追加到前日,「失败记录.xlsx」预期路径不存在
    2. 修法: RUN_REPORT_DIR 改为每次写文件按 datetime.now() 算(惰性)
    3. 当前: 用户每天首启动重置(走 _init_run_dir)
  待修: memory 「待修清单 ①」,大版本

【19.4 信用分 < 60 群体性下滑】
  现象: 群发 9 篇后整批账号信用分 100 → 55,触发 <60 硬终止
  根因(2026-04-29): 头条首发"取消勾选"假动作扣分(v1101.3 修)
  应对:
    1. 立即停发文 + 等信用恢复(平台一般 7-30 天)
    2. 修代码层: v1101.3 hotfix 真坐标 + cb.checked 回读校验

【19.5 cliclick 落空 / 罐头窗口太小】
  现象: cliclick 物理点击坐标超出窗口边界
  应对:
    1. 缺哥手动最大化罐头窗口
    2. 长期 TODO: 脚本启动前自动检测窗口超界
  历史: 2026-04-29 11:30 neo2 实证

【19.6 守护误抢前台(用户切窗被弹回罐头)】
  现象: 用户切 Terminal/VS Code 1 秒内被弹回罐头
  应对:
    1. ssh 该机 ps aux | grep keep_canned_top
    2. pkill -f keep_canned_top + 验证现象消失
    3. 加白名单 + 防抖 (v2)
  铁律: 部署多次时务必先 pkill,严禁 v1/v2 叠加

【19.7 NOTICE selector 失效(平台 DOM 改版)】
  现象: check_system_notice 调 N 次全空,实际有违规
  应对:
    1. 实测: CDP attach 罐头 webview → 手动跑 selector 看是否命中
    2. DOM 改版 → 重写 selector(参 v1102 NOTICE 重建)
    3. 持久化文件 notice_checked.txt 保证不重读

【19.8 v2 算法历史债 — 第一次启动需手动注入 last_published.txt】
  现象: v1102.2 patch 首次落地,前进程没写 last_published.txt
  应对:
    1. 看 log 找最近 OK 发布账号 + 时间戳
    2. echo "账号|时间戳" > run_reports/<日期>/last_published.txt
    3. 重启 → v2 算法读到 → 正确接续
  之后每次 publish 自动写,不再需要手动
