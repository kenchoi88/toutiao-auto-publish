### V1102.6 修正版落地 — 接续算法移出 if + quota cap 对齐铁律

**背景:**
5/9 V1102.6 初版 patch 含 3 笔, 但 patch 2/3 思路错被左右审过驳:
- patch 2 sub_rounds 动态 (=max_quota) 改成 dynamic — 跟 V1102.2 line 211 「微头条 5 篇/天」铁律冲突
- patch 3 去 cap 5 — 直接违反平台硬上限, 风控暴露

绣虎承认错, 修正版 5/10 落地:

**最终 V1102.6 包含:**

1. **接续算法移出 if `_wl_q:` 块** (5 机 × 微头条 + 文章自动)
   - 原: 接续段嵌在 `if _wl_q:` 内 → 白名单空时不接续 → 手动重启从头跑 (NEO2 5/9 实证)
   - 改: 接续段提到 if 外 → 无论白名单是否有都从 last_published.txt 接续

2. **quota cap 对齐铁律** (5 机 × 微头条 cap 5 / 文章自动 cap 3)
   - 原: 微头条 quota = (len(docs) + sent_total) // len(accounts) 没 cap
   - 改: quota = max(1, min(5, ...))  ← 微头条 5
        quota = max(1, min(3, ...))  ← 文章自动 3
   - 文章定时 V1102.5 line 2607 已用 min(3, ...), 不动

3. **撤销** sub_rounds 动态 + 撤销 去 cap 5 (回到 V1102.5 状态)

4. **重置铁律** 写进 V1102.6 版本说明:
   - 用户想全新开跑 = 必须同时清三处:
     - 账号配置.xlsx 「白名单」+「本轮已发」sheet
     - 当日 `运行报告/<日期>/last_published.txt`
     - 当日 `运行报告/<日期>/notice_checked.txt` (可选, 防 NOTICE 误跳)
   - 只清 sheet 不清 last_published.txt → 仍接续 (V1102.6 接续段移出 if 后强化了这个依赖)
