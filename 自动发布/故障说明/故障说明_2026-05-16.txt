  故障 1 — V1102.6 接续算法伪生效 5 天 (sent_accounts_set 预注入跳过接续点)
  ─────────────────────────────────────────────────────────────────
  影响范围: 4 Mac × 微头条 (V1102 落地以来), V1102.6 后所有 mac 真版
  发现时间: 2026-05-16 14:48 mini 重启 + 缺哥实证暴露
  影响累计: 5 天 (5/10 V1102.6 → 5/16) 所有 mac 中断重启的「接续点账号本轮已发部分」场景都伪生效

  现象:
    - mini 14:43 微头条 socket 异常崩, 缺哥重启
    - log 写「[v1102.6] 中断处自动接续:最近 publish「小狗的思念」→ 从下一位「汐云泪」起跑」 ✓ 看起来对
    - 实际首篇 publish: [大1/小1/A] 「生活家居客」 -> ...docx ❌ 不是汐云泪!
    - 缺哥肉眼对比 log 和实际行为, 发现接续算法没真生效

  根因:
    main 函数 line 2589-2594 注入:
      sent_accounts_set = set()
      for _sv in sent_count_map:
          sent_accounts_set.add(_sv)        ← 这两行
    run_death_grip 内 _is_eligible:
      return (name not in dead_terminated and
              name not in sub_skipped and
              name not in sent_accounts_set and    ← 这条 reads 预注入
              acc_count.get(name, 0) < per_account_quota.get(name, 0))

    接续算法 line 2670+ 把 accounts 重排让接续点 (汐云泪) 做首位 ✓
    但 _is_eligible(汐云泪) 检查 `name not in sent_accounts_set`:
    汐云泪本轮已发 2 篇 → 在 sent_count_map → 在 sent_accounts_set → 跳过
    顺位往下找 → 生活家居客 (本轮 0 篇, 不在 sent_count_map) → eligible → publish

  V1102.6 (5/10 NEO2) 当时为什么实证通过:
    左右记忆 project_V1102_6完整_5_10.md line 46 写「验证标志: log 出『中断处
    自动接续:从下一位 YY 起跑』」 — 只看 log, 没看实际 publish 行为。
    缺哥 5/10 实验场景接续点账号 (吧啦啦熊样) 本轮**没发过** → 不在 sent_accounts_set
    → eligible → 实际首篇真是吧啦啦熊样 → 看起来 OK
    今天 5/16 场景接续点账号 (汐云泪) 本轮**已发 2 篇** → 触发跳过 → 伪生效暴露

  修法:
    删除 main 函数 sent_accounts_set 预注入循环 (2 行):
      - for _sv in sent_count_map:
      -     sent_accounts_set.add(_sv)
    保留 sent_accounts_set = set() 初始化 + run_death_grip 内每小轮 add (小轮内防重复)

  实证 (V1103 落地后):
    mini 17:13 关「爬回墙等红杏」publish 中途, 重启:
      log: 「从下一位 爬回墙等红杏 起跑」

  故障 4 — sub_rounds 写死跟 cap 同款坑 (Stage 2 死磕)
  ─────────────────────────────────────────────────────────────────
  影响范围: 4 Mac + 台机 × 微头条 / 文章自动 / 文章定时 Stage 2

  现象:
    - 微头条 mac: sub_rounds=5 / win: sub_rounds=10 写死

    quota=8 但 sub_rounds=5 → 每号死磕 5 圈 → 漏 3 篇/号

  根因:

  修法 (V1103):
    sub_rounds = _quota_per (动态对齐 quota)
    函数签名默认值改 sub_rounds=1 占位

  ─────────────────────────────────────────────────────────────────
  影响范围: 4 Mac × 微头条 / 文章自动 / 文章定时 (有白名单时)

  故障 9 — 台机 collect_accounts 漏 64 个账号 (304 → 240)
  ─────────────────────────────────────────────────────────────────
  影响范围: 台机 × 微头条 / 文章自动 / 文章定时 collect 阶段
  发现时间: 2026-05-16 18:53 缺哥测试文章定时启动暴露

  现象:
    缺哥实际罐头侧栏 304 个账号
    脚本 collect_accounts 出 240 个 → 漏 64
    手动滚到底再启动 → collect 304 ✓
    → 推断: 罐头侧栏虚拟滚动 lazy render 中, 脚本提前结束读取

  根因:
    V1102.8 win 简化 collect_accounts (`if before_top == after_top: break`)
    罐头侧栏虚拟滚动 lazy render 中, scrollTop 暂时不变 → 误判到底 → break
    兜底「滚到底再扫一次」只补 20 个 (5/13 落地的兜底), 不够 304 场景

  修法 (V1103 B 方案):
    主循环加 same_count 容错 (连续 4 次不动才认定到底):
      time.sleep(0.3) → 0.5  # 多等 lazy render
      if before == after: same_count += 1
      if same_count >= 4: break
    兜底循环 (B 方案核心):
      while no_new_count < 3 and bottom_round < 20:
        滚到底 + sleep(2.0) + 扫
        if 新增 → no_new_count = 0
        else → no_new_count += 1

  实证 (V1103 落地后):
    台机 18:56 启动: collect 出 304 个账号 ✓
    早窗 304 + 中窗 304 + 晚窗 304 = 共 912 任务 ✓

  代价: collect 时间 ~15s → ~40-60s, 缺哥拍「慢就慢」, 稳定优先

  防再犯:
    立 memory feedback_collect_accounts_B方案_稳定优先.md
