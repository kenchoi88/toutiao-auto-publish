一、本次唯一改动 — 5s 自动补点
============================================================

【1.1 背景】
  详见 故障说明_2026-05-05.txt 同期条目。
  简要复述:
    文章端 publish_article 里 cliclick(mac) / win32api(台机) 物理点击
    「预览并发布」按钮第一次常无效,等"确认发布"30s 不出 → 进 attempt 2 重试
    单篇耗时拉到 ~2 分钟才发出去
  实证:5 机文章端历次累计 ~7163 次重试(文章自动)/ ~739 次(定时),长期常态
  根因:React/Vue 渲染时序的 "DOM 已出现但 click handler 未绑定" 窗口期
  反风控约束:必须物理点击保留 isTrusted=true,不能用 JS .click() 绕过
  → 只能靠重试兜底缩短等待

【1.2 改法】
  publish_article 里等"确认发布"按钮的 60 次循环 (60 × 0.5s = 30s):
    原:静等 30s 不出 → 走 attempt 2 重发预览并发布
    改:循环到 i==10 (=5s) 仍未见确认发布按钮 → 立即原地重发一次
       「预览并发布」点击命令
       mac:    subprocess.run(["cliclick", f"c:{preview_x},{preview_y}"], ...)
       台机:   win32api.SetCursorPos((preview_x, preview_y))
              + win32api.mouse_event(LEFTDOWN/LEFTUP)
  等价于把今晚 (2026-05-05 17:19~19:51) 在 NEO 上跑过的独立守护脚本
  逻辑内嵌进主进程,主脚本自给自足,不依赖外部守护。

【1.3 效果预期】
  单篇 cliclick 失效场景的等待时间从 30s 压到 5s (-83%)
  总篇耗时从 ~2 分钟级 → ~30 秒级
  不影响 cliclick 一次成功的篇 (5s 内自然推进确认发布,不触发补点)
  守护脚本临时验证(NEO 17:19~19:51): 救援 62 次 / 0 篇真失败 / 单篇等待
  从 ~2 分钟 → 12 秒,验证可行

【1.4 不破坏】
  - 5 次 attempt 外层上限保留 (极端情况平台真不响应,依旧走完)
  - JS 找按钮坐标逻辑不变
  - cliclick m + sleep 0.5 + c 时序不变 (反风控时序)
  - 反风控 isTrusted=true 不变 (仍物理点击)
  - 微头条端代码不动
