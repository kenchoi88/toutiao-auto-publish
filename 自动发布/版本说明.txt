v1101 — 2026-04-28(基线版本)
============================================================

【功能清单】

  一、账号管理
    ▸ 启动时收集罐头侧边栏所有账号(含虚拟滚动兜底)
    ▸ 滚动收集失败 → 滚到底再扫一次补漏(lazy render 兜底)
    ▸ 白名单优先,白名单为空时全收
    ▸ 永久跳过 / 排除账号过滤

  二、文档导入
    ▸ cliclick / win32 物理点"文档导入"按钮
    ▸ NSOpenPanel(Mac)/ Win32 GetOpenFileName(Win)填路径打开
    ▸ macOS 26 主对话框不自动关 → cliclick "打开"按钮兜底
    ▸ 字数检测取最长 ProseMirror 元素(避开标题输入框 placeholder)
    ▸ 字数 < 50 → 重试 fill_dialog 一次,救空文档冤杀

  三、账号查找(底部账号兜底)
    ▸ 滚动找账号
    ▸ 滚动找不到 → 用搜索框输入账号名过滤(底部 lazy account 救命招)
    ▸ 找到后 click → wait → 取 webview ws_url

  四、罐头窗口管理
    ▸ ensure_gtg_top 强化:最小化 Code/Safari 等遮挡者 + 取消罐头自身最小化
    ▸ unhide(处理 Cmd-H Hidden 状态)
    ▸ AXRaise 强制 reorder
    ▸ verify frontmost(失败重试 3 次)
    ▸ Win 飘屏外坐标 → SetWindowPos 拉回(只 win)

  五、点击重试
    ▸ cliclick 文档导入按钮 30s 内弹窗未出 → 重 activate + 重读坐标 + 再点(共 3 次)
    ▸ 预览并发布按钮 30s 内确认弹窗未出 → 第 2 次预览(封面图慢的兜底,最多 5 次)
    ▸ [DIAG] 抓"确认发布"按钮 DOM dump(数据驱动后续选择器优化)

  六、账号状态检测
    ▸ 失登:请登录 / 登录已失效 / 账号已下线 / 重新登录
    ▸ 封号:账号已被封禁 / 账号异常 / 账号被封
    ▸ 禁言:账号被禁言 / 发言受限 / 无法发布
    ▸ 限流:操作频繁 / 请稍后再试
    ▸ 4 类硬终止 → 永久放弃该账号 + 写终止表

  七、信用分 + 头条首发
    ▸ 读信用分(JS 抓 .credit-score 等)
    ▸ 信用 ≥ 95 + 不在 NOFIRST_ACCOUNTS → 自动勾"头条首发"
    ▸ 否则取消勾选

  八、发文流程
    ▸ 微头条:点"文档导入" → 选文件 → 字数/信用/首发 → 直接发布 → OK
    ▸ 文章:点"文档导入" → 选文件 → 字数/信用/首发 → 预览并发布 → 确认发布 → OK
    ▸ 文章定时:同文章 + 设置定时日期 + 排到云端

  九、阅读量回查 + 高阅读告警
    ▸ 发文成功后,导航到数据概览,读最近 3 篇阅读量
    ▸ 任一 ≥ 5000 → 写"高阅读量提醒.txt"

  十、熔断机制
    ▸ Stage 1:连败 6 次 → 停 + Sosumi 系统通知
    ▸ Stage 2 死磕:连败 6 次 → 停 + Sosumi 系统通知

  十一、报表
    ▸ 运行报告/YYYYMMDD/运行日志.txt
    ▸ 失败记录.xlsx
    ▸ 高阅读量提醒.txt
    ▸ 汇总报告.txt(收尾)

v1101.2 — 2026-04-29 凌晨(末位账号 CDP 注入彻底修复)
============================================================

【修了哪些 bug】

  Bug 1 — 末位账号搜索框 fallback 用 cliclick + osascript keystroke 失效
    现象:罐头侧边栏底部账号 cliclick (108,725) 不触发 webview 切换,partition 永远找不到
         scroll_find_account 滚动失败后 fallback 到搜索框,但用 cliclick 点搜索框 +
         osascript keystroke 输入账号名,在罐头不在前台 / VSCode 占着时不工作
    影响:Air 海屿星辰一个账号 04-28 凌晨 5+ 大循环死磕 webview 找不到,5 篇文档堵死
    修法:scroll_find_account 搜索框 fallback 段把 cliclick + osascript keystroke
         整体替换成 CDP JS 注入(neo 同款):
           var setter = Object.getOwnPropertyDescriptor(
                          window.HTMLInputElement.prototype, 'value').set;
           setter.call(s, 账号名);
           s.dispatchEvent(new Event('input', {bubbles:true}));
           s.dispatchEvent(new Event('change', {bubbles:true}));
         直接调 React/Vue controlled input 内部 setter + dispatch input/change,
         前端搜索过滤认作用户输入。完全不需要罐头在前台,VSCode 占着也不影响。
    清空搜索框同款替换:setter.call(s, '') + dispatch input/change

【实战印证】
  04-29 01:06 阿良 CDP 实时注入到运行中的罐头(browser-level Target.attachToTarget,
  不挤主脚本 page ws):
    - 注入前: items.length=N, cliclick 落 (108,725), webview 不响应
    - 注入后: items.length=1, cliclick 落 (108,459), webview 一次匹配成功
    - 主脚本接连发出海屿星辰剩余 5 篇,295→300 全发完(连续 4 篇 (108,459) 一击命中)

【影响】
  - 末位账号 / 任何 cliclick 不响应位置的账号都能被搜索框过滤救活
  - 搜索框 fallback 不再依赖罐头前台,后台可发
  - 取消对 osascript keystroke 的依赖,跟 VSCode / 输入法 / 其它前台应用解耦

v1101.1 — 2026-04-28 下午(熔断重构)
============================================================

【修了哪些 bug】

  Bug 1 — 旧熔断不分原因一刀切退脚本
    现象:任何失败连续 6 次 → consecutive_fail 触发 break + Sosumi → 整个脚本退出
    问题:6 个不同原因的失败也熔断,信用低 / 网络抖动 / cliclick 偶失全混算,
         导致脚本经常半路退出,夜里没人重启 = 大量未发文积压
    修法:取消 consecutive_fail 熔断,改成账号级分类处理(见下 6 条)

【新熔断/重试机制(6 条)】

  ① 禁言 / 封禁 / 信用分 < 60
     → 一次命中立即放弃,记入「问题账号」,不再进大循环
     (信用分 < 60 是新增硬终止类别,以前不算,现在跟禁言/封禁同等对待)

  ② 失登
     → 每小轮 Phase A + Phase B 各 1 次(小轮内 2 次机会)
     → 大循环结束仍失登 → 记失败,不进 Stage 2 死磕
     (避免 Stage 2 死磕一直撞失登浪费时间)

  ③ 普通账号失败(非失登/封禁/低信用)
     → 每小轮 2 次(同失登节奏)
     → 大循环结束 → 进 Stage 2 死磕循环

  ④ Stage 2 死磕节奏 = 一轮过所有账号 1 次,不连续狂磕同账号

  ⑤ 取消旧 consecutive_fail 6 连败熔断 — 用 ① ② ③ 替代

  ⑥ Stage 2 退出条件 = 仅「所有账号 quota 全满」
     其它情况(包括连败、网络断、罐头响应慢)都继续死磕,绝不熔断退出

【影响】
  - 不再有"夜里 6 连败熔断退脚本"现象
  - 失登账号给 N 次机会,跨小轮恢复机会
  - 信用低账号一刀切剔除,避免低质量发文
  - Stage 2 死磕到 quota 满才退,中间任何故障都不退(配合账号级 skip 防 stuck)
