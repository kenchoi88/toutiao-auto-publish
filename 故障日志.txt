  var s = document.querySelector('input[placeholder*="账号"],input[placeholder*="手机"]');

1. catchup.py 服务的是 "timer 漏 → batch 接力补发" 场景,**没覆盖 "timer 中断 → 重启 timer 自己接着跑"**

导致重启后:
- sheet 空 → acc_count 全 0
- 主循环跑满 N 小轮(每号发 N 篇)
- 加上中断前已发的 M 篇 = **M+N 超 quota**

【缺哥拍板的业务语义(2026-05-04)】
1 次发文 = 1 大循环 = N 小轮(N = 素材数 // 账号数,动态算)
1 小轮 = 全账号各发 1 篇
中断在第 X 轮第 Y 账号 → 重启从该轮该账号继续:
  - 上面账号(< Y):已发 X 轮 = sheet count=X = 跳过
  - 中断点 + 下面(≥ Y):已发 X-1 轮 = sheet count=X-1 = 还差 1 篇
  - 排第 X 小轮:101 个账号(中断+下面)各发 1 篇 → 全员齐活进下一轮
"本轮不重发 + 缺 N 补 N" = 业务规则推出来的"从中断处起"最简方案

【v1102 修补】
1. 「本轮已发」 sheet 加 **「已发次数」列** (账号, count)
2. sheet **不在小轮末 clear**,**累积到大循环末才 clear**(齐活才清)
3. 每次成功发文 → sheet 该账号 count += 1
4. 主线启动 → 读 sheet 重建 acc_count[账号] = count → 算 quota - count = 缺 N
5. 主循环 acc_count < quota 自然停 = 不超不漏

【2026-05-05 故障 · 文章端 cliclick 首次点击预览并发布常失效】
谁记的: 绣虎(Win 台机) + 缺哥定位
状态: ⏳ 待 V1102.1 衍生 修复(5s 自动补点),代码尚未改

【现象】
  文章自动发布 + 文章定时自动发布:
    cliclick(mac) / win32api(台机) 物理点击「预览并发布」按钮,
    第一次常无效 → 等"确认发布"按钮 30s 不出 → 进 attempt 2 重试
    单篇耗时拉到 2 分多才发出去
  微头条端不存在此问题(无两步弹窗)

【实证(5 机历次累计 "确认发布未出现" 重试次数)】
  文章自动: NEO 1619 + mini 2021 + neo2 1856 + air 1462 + 台机 205 ≈ 7163
  文章定时: NEO  116 + mini   43 + neo2  366 + air   30 + 台机 184 ≈  739
  → 主要集中在「文章自动」(~10 倍于定时)
  → 台机(win32api) 4-5 倍优于 mac(cliclick),但仍存在
  → 长期常态,非昨日突变

【根因】
  React/Vue 渲染时序的"DOM 已出现但 click handler 未绑定"窗口期:
    JS getBoundingClientRect 返回坐标 = DOM 已渲染
    立刻物理点击 → 落到刚渲染但还没绑定监听的按钮 → 点空气
    几百毫秒到几秒后 handler 绑定完成,下次点击才生效
  「预览并发布」触发预览渲染(封面图加载、文档预览),窗口期更长
  「定时发布」按钮弹时间设置框,UI 流程不同,窗口期短

【为何不能用 JS .click() 绕过】
  头条平台反爬检测 event.isTrusted = false 即识别为脚本 → 影响推荐
  必须物理点击保留 isTrusted = true
  所以只能靠重试兜底,无法根除
