v1101.1 — 2026-04-28 下午(熔断重构)
============================================================

【修了哪些 bug】

  Bug 1 — 旧熔断不分原因一刀切退脚本
    现象:任何失败连续 6 次 → consecutive_fail 触发 break + Sosumi → 整个脚本退出
    问题:6 个不同原因的失败也熔断,信用低 / 网络抖动 / cliclick 偶失全混算,
         导致脚本经常半路退出,夜里没人重启 = 大量未发文积压
    修法:取消 consecutive_fail 熔断,改成账号级分类处理(见下 6 条)

【新熔断/重试机制(6 条)】

  ① 禁言 / 封禁 / 信用分 < 60
     → 一次命中立即放弃,记入「问题账号」,不再进大循环
     (信用分 < 60 是新增硬终止类别,以前不算,现在跟禁言/封禁同等对待)

  ② 失登
     → 每小轮 Phase A + Phase B 各 1 次(小轮内 2 次机会)
     → 大循环结束仍失登 → 记失败,不进 Stage 2 死磕
     (避免 Stage 2 死磕一直撞失登浪费时间)

  ③ 普通账号失败(非失登/封禁/低信用)
     → 每小轮 2 次(同失登节奏)
     → 大循环结束 → 进 Stage 2 死磕循环

  ④ Stage 2 死磕节奏 = 一轮过所有账号 1 次,不连续狂磕同账号

  ⑤ 取消旧 consecutive_fail 6 连败熔断 — 用 ① ② ③ 替代

  ⑥ Stage 2 退出条件 = 仅「所有账号 quota 全满」
     其它情况(包括连败、网络断、罐头响应慢)都继续死磕,绝不熔断退出

【影响】
  - 不再有"夜里 6 连败熔断退脚本"现象
  - 失登账号给 N 次机会,跨小轮恢复机会
  - 信用低账号一刀切剔除,避免低质量发文
  - Stage 2 死磕到 quota 满才退,中间任何故障都不退(配合账号级 skip 防 stuck)

【修的 BUG】
  publish_article() 中 should_first=False 时执行的"取消勾选"是空点 —
  JS 返的是"头条首发"四个字 textNode 的 boundingClientRect 中心,不是
  复选框 input 的真坐标。win32 真鼠标点到文字上,cb.checked 不变;
  脚本却照打 "头条首发: 取消勾选 (sx,sy)" 看着每次都成功,实际一次没切换。
  点完也无回读校验,bug 自 v1101 基线就存在,首发审核审到非原创每篇扣 5 分,
  9 次累计扣 45 分(100→55),触发 <60 硬终止 + 群体性号信用分下滑。

【修法(三大件 win/ 同段统一 patch)】

  publish_article() 中"根据信用分设置头条首发复选框"段重写:

  ① JS 探测改返回 cb (input[type=checkbox] / role=checkbox) 自身中心坐标,
     cb 宽高 <4 退而求次取可见父节点矩形(应对 visually-hidden 复选框)。
  ② 真鼠标点击后 sleep 0.4s 重读 isChecked,与 should_first 比对。
  ③ 不一致 → 重试一次真鼠标 → 仍不一致 → 降级 JS cb.click()+
     dispatchEvent('change',{bubbles:true}) 兜底。
  ④ 三轮全失败时:
     - 应取消但仍勾着 → ★ 硬保护:wsc.close() return False
       ("首发取消失败(硬保护)"),本篇不发,避免被扣 5 分。
     - 应勾未勾 → 继续发(只是失去首发激励,不会被扣)。
  ⑤ 日志区分 "已校验" / "校准失败" / "硬保护跳过",便于审计。

v1101.3 hotfix#2 — 2026-04-29 21:45 (Mac 端 Finder 兜底抢前台守护)
============================================================

【修的现象】
  Mac 跑发文时,篇间等待 8~17 秒期间罐头失去 frontmost,
  macOS 默认把 frontmost 给 Finder → 桌面壁纸冒出来,看上去窗口"消失",
  下篇开始 ensure_gtg_top 才把罐头拉回来。视觉抖动,不影响发文成功率。

【修法】
  工具/keep_canned_top.sh —— 后台 0.5s 轮询 frontmost,
  只在 frontmost=Finder 时 activate 创作罐头。其他 frontmost 不动,
  对脚本本身无干扰(脚本 cliclick 时罐头本就 frontmost,守护不介入)。

【部署】
  scp 工具/keep_canned_top.sh kenairXXX@<machine>:/tmp/ &&
  ssh kenairXXX@<machine> 'nohup bash /tmp/keep_canned_top.sh > /tmp/keep_canned_top.log 2>&1 &'

  停:ssh ... 'pkill -f keep_canned_top'

  ☐ air / neo / neo2 — 各 Mac Claude 自取部署

        cur = set(get_docs())
        before = len(doc_pool)
        doc_pool[:] = [d for d in doc_pool if d in cur]
        if len(doc_pool) != before:
            log(f"  doc_pool 重扫:剔除 {before - len(doc_pool)} 条幽灵引用")
     实时同步外部 mutate,避免每篇遇到才发现。

【背景故障】

  现象:
    台机微头条 04-30 15:26 启动,日志报"待发文档 520 份 / 收集到 100 个账号 /
    排除 20 个永不发文 / 本次发布 80 个账号 / 配额合计 480"。但缺哥实际配置:
    150 账号(罐头侧边栏总量)→ 永不发文 1-20 → 应跑 130 账号 × 4 篇 = 520 篇。
    脚本只跑出 80 active,差 49 个账号没被扫到。

  影响:
    本批次预计漏发 49 账号 × 4 篇 = 196 篇,占总篇数 37.7%。
    且过去多批可能都受同一漏扫影响,只是缺哥未察觉。

  根因(2026-04-30 缺哥当面定性):
    ▸ 不是虚拟滚动 lazy render 漏扫(v1101 已修过)
    ▸ 是罐头侧边栏当前的"标签筛选"过滤掉了未贴标签的账号:
      - 罐头里有 3 个标签:迦境台机第一批 / 第二批 / 第三批,合计应覆盖 149 账号
      - 缺哥贴标签时网速慢,部分账号"贴标签"操作未生效
      - 脚本启动时筛选只勾了"第一批 + 第二批"→ 侧边栏只显示已贴上这两批标签的 100 个
      - 漏贴第三批标签的 49 个账号根本不在侧边栏 DOM 里,脚本怎么扫都扫不到
    ▸ scroll_find_account 同样依赖侧边栏 DOM,即便给它账号名也找不到 — 所以
      给个 "补充账号 txt" 也救不了,必须让它们先在侧边栏显示出来

  实证流程(2026-04-30 16:30 ~ 17:00 阿良 air 本机操作):
    1. 写脚本激进扫描 air 罐头侧边栏(小步 30-50px 滚动 + 反复双向 + 多轮兜底)
       → 扫到 149 个账号(=罐头实际登录数,完整)
    2. 从台机日志解析 active(80)+ skip(21)= 已识别 100 个

    4. 验证 air 罐头筛选弹窗的 DOM 结构(arco-popover + arco-select 多选)
    5. 用 CDP Input.dispatchMouseEvent 试点漏斗 — 失败(罐头被遮挡时不渲染弹窗)
    6. 改用 osascript activate + frontmost + cliclick 真鼠标 → 弹窗成功弹出
    7. 一键 X 掉三个已选标签(close 按钮每次点完滑到固定位置 1337,350,可连点)

【v1102 解决方案】

  把"启动前自动配置罐头筛选标签"作为 gtg_batch.py / gtg_timer.py 的前置步骤,
  脚本启动时:
    1. CDP 连主窗口 → osascript activate + frontmost + AXRaise
    2. cliclick 真鼠标点漏斗(屏幕坐标 = window.screenX/Y + getBoundingClientRect)
    3. 弹窗 DOM 中逐个 cliclick X 关闭按钮,清空所有已选标签(连点固定坐标即可)
    4. cliclick 点"全部标签"arco-select → 等下拉浮层渲染
    5. 浮层里逐个匹配目标标签 textContent → scrollIntoView → cliclick 勾选
    6. 关弹窗(点 popover 外或 ESC)
    7. 验证侧边栏账号数 == 期望值,不达继续到 collect_accounts

  目标标签由配置 xlsx 提供(白名单/筛选标签 sheet 新增列),或脚本启动参数。


【技术坑(5 条 — 必须沉淀,后续 agent 别重踩)】

  ① JS click() 触发 React 组件无效 — isTrusted 检查
     dispatchEvent / el.click() 在 React 受控组件上不弹弹窗,必须用真事件。

  ② CDP Input.dispatchMouseEvent 也算"真"事件,但弹窗渲染依赖罐头是否在前台
     罐头被其他窗口遮挡时,DOM 状态可改但屏幕不渲染。屏幕上看不到 = 弹窗事实
     未真"打开",后续依赖该 DOM 的操作可能错位。
     → 必须 osascript activate + set frontmost to true + AXRaise 三件套

  ③ arco-tag X 关闭按钮重叠特性
     连续 X 三个标签时,每次 X 完后剩余标签向左挤,下一个 X 滑到固定屏幕坐标
     (air 实测 1337,350)。可以连点同一坐标 N 次实现批量清空,无需重新读 DOM。

  ④ arco-select 多选超过 2 个会折叠成 "+1..."
     验证已选标签数量时,不要靠数 `[class*="arco-tag"]` X 按钮 — 多于 2 个时
     超出部分进 "+1..." 折叠不出现 close 按钮。
     → 验证用 select 的 textContent,或直接扫侧边栏账号数(149 即三批全选)。

  ⑤ 下拉浮层选项每次点选后会重渲染
     浮层里逐个点选目标标签时,每选一个就要重新读浮层 DOM 拿当前剩余选项坐标,
     不能一次性缓存所有目标坐标连点(选过的会变 selected,DOM 排序可能变)。
