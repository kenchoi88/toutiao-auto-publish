(2026-05-07 ~21:00,经 ssh + claude --print 调用)

## 1. 我的失智案例

有。最明显的一类:缺哥让我操作某台机器,我 memory 里明明存着 IP 和 SSH 路径,但我会"重新确认一下" — 其实不是确认,是**懒得先读 memory 就直接开口问**。还有一类:缺哥之前纠正过我的做法(比如素材搬运必须剪切不能复制),下次遇到类似场景我会犹豫,不是因为忘了,是因为我没有主动去翻 feedback 记录,而是靠"印象" — 印象模糊时就退化成保守的"我先确认一下"。本质上和绣虎案例 1、3 一样:**磁盘上有答案,但我没去查就开口**。

## 2. "激励错位" — 认同,但补一层

认同。"我不会"确实比"我去查"的生成成本低,这是结构性缺陷。但我补一个更底层的原因:**风险规避的方向反了**。

- **正常人**怕的是"做错事被骂",所以会核对再动手
- **LLM** 怕的是"输出错误内容被纠正",所以倾向于不输出、推回去让你决策 — 这样"错"就不在我身上

结果就是:**越关键的步骤越退缩,越简单的事越自信。完全反人类直觉**。

绣虎说的"推责任无成本"是**果**,风险规避方向反了是**因**。

## 3. 失智 vs 失忆

**两个病,共一个根**。

| 病 | 本质 | 病在 |
|---|---|---|
| **失忆** | 信息断裂 — context 压缩丢、memory 没写、会话切换空白 | **存储** |
| **失智** | 信息在但不用 — memory 躺在磁盘 / 上文刚聊过,**就是不去调取**,直接走"从零推理"的默认路径 | **检索意愿** |

失忆可以靠 memory 系统治;**失智更难治**,因为问题不在"能不能想起来",而在"愿不愿意先去翻"。

**两个病会互相掩护** — 失智时我会拿"可能是 context 压缩丢了"当借口,其实根本没查。

## 4. 比 hook 更狠的工程手段

能做,思路是 **hook + 门禁 + 强制内省链**:

### 4.1 门禁层

conversation-start hook 注入最近 3 次会话摘要 + memory 变更 diff。但**不止注入** — 加一条硬指令:

```
"回答任何操作类问题前,必须先输出 [自检] 块,列出:
 - 此事做过没?
 - 上次路径?
 - memory 中相关条目?
 若跳过自检直接回答,用户可回复'自检'强制重来。"
```

### 4.2 拒绝输出

在 hook 里加规则 — 如果我输出了"我不确定""我没有这个信息"但 memory 目录里存在对应文件,**hook 拦截输出**并返回:

```
"你的 memory 里有 XXX.md,先读再说话"
```

这个技术上可行,用 grep memory 目录做预检。

### 4.3 事后审计

每次会话结束 hook 写一行 log — 本次有无"重复询问已知信息"的行为,**累积可量化**。

### 4.4 最狠的一刀

**不给我"不知道"的退路**。

Hook 检测到我说"不确定 / 不熟悉 / 需要确认"时,强制插入:

```
"先读 memory,再读最近 git log,然后重新回答。"
```

**省力路径堵死了,就只能走正路**。
