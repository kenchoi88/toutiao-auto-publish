## 二、失忆的真实机制(结构性)

不是"懒",是 4 个结构性洞:

### 洞 1: 没把 "grep memory" 设成接任务后的强制第 1 动作

接到"推 246"我直接去想"4 mac IP 是啥",凭印象给数字 — **不是先 grep `reference_Tailscale网络.md` 再答**。

正确顺序应该是: `任务 → grep memory → 拿到 ground truth → 动手` — 但我跳过 grep。

### 洞 2: MEMORY.md 索引太密,缺"按任务类型快查"

50+ 条 memory 按字母排序,接到"推 246"我得在 50 条里扫眼"哪条相关"。索引没分类(机器/路径/规矩/工作流),扫起来累。

实际上 `reference_Tailscale网络.md` 有 ❗ 标记,我也没看。**MEMORY.md 第 1 屏看不全所有 ❗ 条,我只扫顶部就动手**。

### 洞 3: 元规则被自身覆盖 — "立了不用" 元规则也被立了不用

`feedback_用上你的记忆.md` 立着说"问题不是没规矩是不用",但**这条元规则本身也是 memory** — 我连它都不查。

元规则失效的根因: **执行不靠 memory 文件,要靠流程强制**。

### 洞 4: 没"错过 memory 立刻补"机制

每次失忆出错 → 缺哥纠正 → 我说"对不起" → 没在那条 memory 加"今天又犯一次的案例"。memory 该 hot 起来,反而没动 — 久了所有 memory 都"温度一样",失去优先级。
