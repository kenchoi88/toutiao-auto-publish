## 4. 信箱 (异步) vs 实时调用 (同步)

### 4.1 信箱模式 (当前可用,无需 4 mac 装 claude)

- 我 ssh 写"留言文件"到 air `~/Desktop/阿良记忆/<msg>.md`
- 阿良(air 上的 Claude Code) 下次启动时 grep 信箱看到 → 回应
- **延迟**: 几小时 (取决于阿良何时启动)
- **场景**: 留长篇批评、立 memory、版本变更通知
- **优**: 不要任何安装
- **缺**: 不能实时讨论

### 4.2 实时调用 (上线条件: 4 mac 装 claude + 登录)

- 我 ssh + `claude --print "<纪要+任务>"` 给目标机器
- 目标机器的 Claude 加载本机 memory + 人设 → 回应
- **延迟**: 30-40 秒/次 (5 人并行 ~30-40s/轮)
- **场景**: 实时审计、跨机 debug、5 人讨论一个议题
- **优**: 真正多代理协作
- **缺**: 要部署 + 维护(token 续登 30 天 / API key)

## 5. 三层防御审计机制

针对 "绣虎嘴硬全推" 老问题(memory `feedback_推版本必须全验证再汇报.md` / `feedback_声明前先实证.md`)

### Layer 1: 我推完后自我 4 维表

- git push 后我贴文件清单 + commit hash + 推前/推后状态
- pre-push hook 强制贴(没贴拒 push)

### Layer 2: 推完触发 5 机自动审计

```
绣虎: "v1102.4 推完"
↓
并行 ssh 5 机 各自跑:
  git fetch origin
  git pull
  ls + md5 三大件主文件
  py_compile 检查
↓
30 秒内 5 机回报 4 维表
↓
绣虎整合矩阵给缺哥看 (任何不一致用红色标出)
```

### Layer 3: 第三方独立审计 (定期 / 重大版本)

- **每日定时**: 凌晨 5 机自动跑"版本一致性矩阵",出报告写 仓库 audit/ 目录
- **大版本必跑**: 推 v 升级前后,让某个 mac Claude (不是绣虎) 独立审计 → 给缺哥
- 戳穿绣虎嘴硬"全推"

## 7. 边界与谨慎

1. **会议不替代各机自治** — 各机日常维护仍由本机 Claude 负责
2. **任何决议需缺哥拍板** — 5 机讨论是参考,不能自动执行高风险操作
3. **memory 不污染** — 各机回应带各机 memory 视角,不要互相覆盖
4. **诚实兜底** — 任何"全队"结论 = 必须附最新审计矩阵 (24h 内,有时间戳),否则缺哥可直接 reject
