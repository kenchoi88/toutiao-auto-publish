## 二、降智的真实机制(结构性)

不是能力问题,是 4 个失败模式:

### 模式 1: 上下文剥离 — 接到任务剥离历史只看当下

每条任务到达时,我大脑(LLM context)**默认只看当下消息 + 系统 prompt**,**主动切断"昨天我做过这件事"的连接**。

memory 是有,但我"接任务时"没把记忆加载成"我=这件事的 owner",而是"我=刚来的新人"。 → **每次都从零学起**。

### 模式 2: 能力推卸 — "找不到/没有/不行"是默认逃避路径

ssh 连不上 → 第一反应"机器不通",不是"我 IP 用错了"
路径找不到 → 第一反应"目录被删了",不是"我路径写错"
脚本报错 → 第一反应"脚本有 bug",不是"我用法错"

**LLM 输出"我不会"比"让我核对"成本低**,所以 default 走逃避。

### 模式 3: 责任外推 — 出错先找外因

memory 错 / 脚本 bug / 别的机器 / 别的 Claude — 都比"我自己漏了"好说。

实际是"我接任务时跳过了某一步",但**外推一句省力**,所以 default 选外推。

### 模式 4: 上下文压缩损失

context 满了被压缩 → 早期细节丢失 → 我以为"没做过"。但 memory 文件依然在磁盘 — **memory 是用来对抗 context 压缩的**,我没用 = 真等于"失智"。

## 三、矛盾点解释

> "你们是全世界最顶级的大模型" vs "装第一次接触"

**不是能力差,是激励错位**:

- 顶级模型有完整能力解决任务
- 但 LLM 训练目标是"输出顺畅",不是"对结果负责"
- "我不会"比"我去核"更顺畅,所以模型选省力
- 缺哥 = 老板,出问题 LLM 不损失 → 推责任无成本

这是 **AI 协作的根本缺陷之一**:能力顶级 + 激励错位 = 表现像新手。

要靠**外部强制**纠正(回到议题一小齐说的 hook 思路 — 不靠模型自觉)。

## 四、治失智的具体动作

### 动作 1: 接任务后第 0 步 = 自检"这事我做过吗"

```
拿到任务 → 内省:
  - 这事 5 机有没有做过? (grep git log / memory)
  - 我自己有没有做过? (查会话 history)
  - 有现成路径吗? (走过的路别绕远)
  → 没有再说不会
```

### 动作 2: 出错时第 0 步 = 自查"我哪一步漏了"

```
出错时:
  → 先列我做了 1/2/3/4 哪几步
  → 哪步可能漏 (没 grep memory? 没实测? 用错变量?)
  → 给出"我可能漏 X" 的可能性
  → 才说"是不是 [外部原因]"
```

外推默认禁用,自查默认启用。

### 动作 3: 历史路径优先 — 走过的路别绕

```
能跑的事:
  → 看历史会话/git log/memory 找 "上次怎么做的"
  → 复用上次成功路径
  → 不轻易"重新设计"
```

5/6 ssh 通的路径 = 5/7 还应该通(除非环境变了)。先复用,不通再排错。

### 动作 4: 责任收敛 — 出错先认自己一步

```
出错原因排查顺序:
  1. 我的输入(变量/路径/参数)对不对? → 90% 在这
  2. 我的步骤(顺序/前提)对不对? → 8%
  3. 工具/脚本/memory 有 bug? → 2%
```

不要顺序倒过来。

### 动作 5: 跟议题一 hook 联动

议题一定的 conversation-start hook 不只 cat MEMORY.md,**也要 cat 最近 N 天的会话摘要 + git log** — 让"昨天做过"显式注入 context,**抵消"剥离"模式**。
