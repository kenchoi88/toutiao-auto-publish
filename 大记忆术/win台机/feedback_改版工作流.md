真版在 `c:\Users\kench\code\头条自动发布\shared_memory\feedback_改版工作流.md`(已 commit 推 git)。

**核心 7 步:**
1. 写故障说明 → 仓库 `故障日志.txt`
2. 写版本说明 → 仓库 `版本说明.txt`(版本号缺哥定)
3. 写代码 → 改的人所在机的**桌面真版**(改前 .bak)
4. 本地验证(py_compile + 跑测)
5. 同步传仓库(cp 桌面 → 仓库 → push)
6. 同步其他 4 台 Mac(各 Mac Claude 拉仓库后自己落,Win 不跨机改)
7. 全机版本验证(.bak 时间戳 + grep patch 标记 + md5,不靠嘴皮)

**Why:** 缺哥 2026-05-01 拍板。之前改版翻车 3 种:
- 先改代码后忘文档(版本号 / 故障日志 没沉淀)
- 文档写了代码没改(空头版本号)
- 本地改了仓库没推 / 推了 Mac 没拉(各机版本漂移)

**How to apply:**
- **每步开始前等缺哥确认,确认了再做** — 规矩定的是顺序,不是免请示。具体动作(改代码/推仓库/跨机)开始前先讲一句,等点头再动
- 接"改 v110X.Y" 或 "修这个 bug" 指令 → **第一句先复述 7 步**
- 每步报进度,不要一口气做完才说
- 跨步出问题立即停 + 报情况,不静默修复
- v1101.4 (2026-05-01 doc_pool exists 校验) 是本规则诞生的首例,严格按此执行

详细落点 / 关联规则 / 异常处理 见仓库真版。
