## 硬规则(不靠承诺,靠机制)

### 1. 推 patch 后强制 4 维实证表

每件单独列,任何一格少 = 不准说"全推完":

| 件路径 | grep 关键标记(列出 N 个) | py_compile | 启动实测 log |
|---|---|---|---|
| 机A/微头条/gtg_batch.py | LAST_PUBLISHED_FILE:6 / 中断处自动接续:1 / ... | OK | log "中断处自动接续:..." 实证 ✓ |
| 机A/文章/gtg_batch.py | ... | OK | ... |
| 机A/文章定时/gtg_timer.py | ... | OK | ... |
| 机B/... | ... | ... | ... |

不贴这表的"全推完" / "全队对齐" / "同款" 一律算虚假。

### 2. 禁用模糊措辞(自动红灯)

下列措辞 = 必须先贴 4 维表:
- "应该 / 大概 / 估计 / 跟微头条同款 / 推全队了 / 全件 / 都装了 / 一致 / 同步 / 完整"

### 3. 版本号一律 V1102 — 不再叠 .1 / .2 / .3

2026-05-05 起新规:
- 后续任何修补 / 新功能 / hotfix 都**在 V1102 大版本下**,不另起 v1102.X 子版本
- 新功能/规则**写到 V1102 版本说明.txt + memory**,不靠版本号区分
- V1102 = 唯一活动版本号,以后再出大改才上 v1103

历史已有的 v1102.1 / v1102.2 标记不动(代码里已经写了,作为内部演进痕迹保留),但 commit / 仓库目录 / 故障日志条目今后不再叠子版本号。

### 4. 推版本流程加一步"功能实证"

改版 7 步工作流(见 feedback_改版工作流.md)第 7 步"全机版本验证" 强化:
- 不只看 .bak 时间戳 + grep patch 标记 + md5
- **必须**实测启动一轮看 log 出现关键标记(如 "中断处自动接续:..." / "从 notice_checked.txt 恢复 X 个")
- 至少 1 件 1 机端到端实测通过,再声明"全推完"

## How to apply

- 接到"推 patch 全队"任务 → 第一动作准备 4 维表模板(列件 × 列标记 × 列 py_compile × 列实测)
- 推完每件**逐件**填表,不准用"假设跟微头条一样"
- 表格全绿才说"全推完",否则只说"已推 N/M 件,还差 X 件 [具体清单]"
- 任何模糊措辞 → 红灯停 → 贴具体数据
- 「声明前先实证」+ 本条配套使用,缺一不可
