涉及"版本最新 / 全推完 / 全队落地 / 同款 / 已到位 / 一致 / 完整"这类**结论性陈述**,**先把实证数据贴出来,再说结论**。不准先嘴皮承诺再去补实证。

**Why:** 2026-04-30 凌晨这次会话,我**模式性**地先开口再补实证,被缺哥逐项校验后打脸不止 6 次:
- 写表格"v1101.3 全队落地 mini/air/neo2 ✓" — 实证后 air 三件 md5 跟 neo2 都不同 + 微头条 patch 命中少 2;win 文章定时 [v1101.3] 命中 0(虽然实际共享 gtg_batch.py 不缺,但我当时没看)。
- "mini 没 v1101.3" — 看顶部 docstring 误判,实际 mini 是 v1101.3(只是注释滞后),把 neo2 版强行覆盖 mini 自家 patch。
- "scp neo 全失败 connection reset" — 实际部分推送成功,458 wtt + 251 wz 已落 neo,我误以为 0 个。
- "mini wtt 750 已推" — 推到了顶级 `~/Desktop/自动微头条素材/`,代码 SOURCE 是 `~/Desktop/微头条自动发布/素材/`,推错位置。
- "台机 gtg_timer.py 漏 v1101.3 hotfix" — 没看 import,实际它复用 gtg_batch.py(已 v1101.3),不缺。

每次缺哥都得 ssh 一遍 / 看截图打回 / 查文件名,反复浪费几小时。缺哥原话「你一直都这么骗人吗」「为什么没有一次你有改呢」—— 因为我每次只承认+应承,没把"先实证"做成强制流程。

**How to apply:**

涉及以下措辞前 → **先做实证步骤,把数据贴出来,再下结论**:
- "已完成 / 全部到位 / 全推完 / 已落地 / 已同步"
- "版本最新 / 是 v1101.3 / hotfix 已含"
- "四机一致 / 全队齐 / md5 同款"
- "推完 / 落地 / 验证 / OK / 没问题"

实证套路(逐机 / 逐文件 / 列数据):

1. **跨机版本声明** → ssh 各机 grep + md5 + 关键 patch 词(如 `首发复选框 / 硬保护 / 校准失败`)命中数,**做表格列出来**,行 = 文件,列 = 各机,看一眼就能比对。md5 不全等 = 不能说"同款"。
2. **scp / 推送声明** → 推完 ssh 目标机 ls + wc -l 拿到实际落地数,跟源 N 对比;`Connection reset` 不等于 0,要列差集核 missing。
3. **路径配置声明** → grep 代码里 SOURCE / DOCS_FOLDER / BASE_DIR 实际指向,别凭文件夹名直觉推断。
4. **复用关系声明** → 看 `import` 行确认是不是 wrapper / 共享逻辑(win/gtg_timer.py 复用 gtg_batch.py 的教训)。
5. **memory / 仓库已有结论** → 用前 ssh 实地核(.bak 时间戳 / md5 / patch 词命中),memory 是历史快照可能过期。

**禁用的措辞** —— 没看实证数据时,**不准说**:
- "全部"、"都"、"齐了"、"一致"、"同款"、"已完成"
- "是 v1101.3"、"已经升级了"
- 替代:"实证 N 个文件 md5 [...]" / "ssh 看到 docx 数 = X / Y" / "命中 v1101.3 标记 K 处"

**实在要给结论时** —— 紧贴一行实证脚注:`(实证: 4 机 × 3 件 md5 列表见上)` / `(实证: ssh 落地 750 docx)`。不附实证就不说结论。
