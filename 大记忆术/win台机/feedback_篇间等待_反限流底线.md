`gtg_batch.py` / `gtg_timer.py` 里 **"篇间等待 8~20 秒"** 不是性能瓶颈,是**头条平台反多账号连发频率检测的故意冗余**。任何"sleep 太长可优化"的提议默认拒绝。

**Why:** 小齐(neo 那台 Claude 历史)改过这段 sleep 想加速,**头条直接限流降权**,从 9 号峰值后收益清零(几乎没流量),改回原间隔才恢复。2026-04-30 我又一次提议"篇间等待自适应能省 3-5s/篇" → 缺哥反驳"几乎没流量,等同限流",再次确认这层是底线。

平台风控点:
- 单台设备/罐头实例**短时间内连发不同账号** = 群控机器人特征
- 不仅是"该账号"被限流,**整批共用罐头的账号一起降权**(我的推测,实证是收益断崖)
- 跟单账号 quota(早中晚 5 篇/号)不同,**这是设备级/IP 级 / 罐头实例级的频率检测**
- 即便每号每天才 3-5 篇,**号 A 发完到号 B 发的间隔太短**,头条 backend 也算不正常流量

跟另一条 [真鼠标点击勿换 CDP](feedback_真鼠标点击_勿换CDP.md) 配一对 — 都是反风控冗余,牺牲速度换安全,**不能动**。

**How to apply:**

1. 看到 `time.sleep(8)` / `篇间等待 N 秒` / `cooldown` 这种代码,**默认是反限流设计**,不要建议"压缩"
2. 速度优化只能从**其他维度**:
   - 子进程开销(osascript 起进程)
   - cliclick 后的本地回读 sleep(不是篇间)
   - JS 选项点击间隔
   - webview 重试链
3. 跨会话再有 Claude 提"篇间等待自适应/缩短/检测罐头空闲就下一篇" → **引用本 memory 拒绝**,理由"小齐 9 号事件 = 限流证据"
4. 风控敏感的 3 类设计(归一类),都不可动:
   - 真鼠标点击(cliclick / win32api)— event.isTrusted + 物理轨迹
   - **篇间等待**(本条)— 频率检测
   - 罐头登录账号侧栏的常规切换节奏(不要批量 JS 切号绕)
