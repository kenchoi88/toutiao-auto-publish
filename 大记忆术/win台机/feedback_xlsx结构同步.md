**红线:** 改 .py 代码涉及读/写 xlsx 的新 sheet 或新列,**必须同步给 xlsx 加上对应结构**(空 sheet + 表头列名),不要等用户跑脚本时报错才发现。

**Why:** 2026-04-26 下午我做"三套脚本死磕重构",`gtg_timer.py:_finalize_config` 加了写"待补漏"sheet 的逻辑,但**忘了给账号配置.xlsx 加这个 sheet**。push 完走人,缺哥晚上发文时发现"配置还是老款",连骂三轮才推完。根因有三:

1. **memory「账号配置 sheet 权属:用户手填,脚本只读」误读** — 这条是说**内容**用户填(账号名),不是说**结构**(sheet 名/表头)用户加。结构变更是代码的事,我必须负责。
2. **`.xlsx` 在 `.gitignore` 排除** — git diff 看不到 xlsx,我意识不到代码改动有 xlsx 配套需求。
3. **没跑测试 commit 就 push** — 跑一次就暴露,我没跑。

**How to apply:**

每次改 .py 代码涉及 xlsx 时,在 commit 之前过一遍:
- [ ] 加了 `wb["新sheet名"]` 或 `wb.create_sheet("X")` 这种代码 → **去 xlsx 把这个 sheet 加上**(空表头按代码里 cell(r, c, ...) 的列序写)
- [ ] 加了 `ws.cell(r, N).value` 这种新列 → **去 xlsx 在表头那行 N 列写上列名**
- [ ] 改了已有 sheet 的列序 → **xlsx 的表头列序对齐**
- [ ] **跑一次脚本** 或 至少 dry-run 模式 → 确认结构齐
- [ ] 三大件结构对齐:微头条/文章/定时如果用同一套配置规则,对所有三个目录的 xlsx 同步

**不在领域内的事(仍归用户):**
- 永久跳过加哪些账号 → 用户填
- 白名单加哪些账号 + 配额 → 用户填
- 定时配置 B1 日期 → 用户填(每天改)

**配套结构 vs 数据内容的边界:**
- 结构(sheet 名 / 表头列名 / 列顺序)= **代码的事,我负责同步**
- 数据(每行账号名 / 配额 / 跳过名单)= **用户填,我不动**
