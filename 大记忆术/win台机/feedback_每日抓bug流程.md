**每日 routine:** 缺哥发起后,我执行以下步骤(无需再问):

## 步骤 1 — 抓 4 mac 昨夜 log

SSH 到 mini / neo2 / air(neo SSH 通后加),抓**昨日**(YYYYMMDD)三大件运行日志:
```
/Users/kenchoXXX/Desktop/{微头条/文章/文章定时}自动发布/运行报告/$(date -d yesterday +%Y%m%d)/运行日志.txt
```

注意机器命名(per `feedback_桌面才是真版.md`):
- mini 文章定时 真版在 `/Users/kenchoimini/code/头条自动发布/文章定时自动发布/`
- air / neo2 文章定时 桌面是 symlink → ~/code/

## 步骤 2 — 跨机错误统计

按错误类型 grep 计数:
- `X 发布失败`
- `文档导入弹窗未出现`
- `文档导入后内容为空`
- `未出现确认发布按钮` / `确认发布未出现`
- `文件对话框反复卡住`
- `找不到文档导入按钮`
- `Traceback`
- `Handshake status 500`(CDP target 错)
- `硬终止`

**报告样式**(per `feedback_重点用大标题.md` 用大标题):
```
## YYYY-MM-DD 跨机错误汇总
| 机器/脚本 | 成 | 失 | 硬终止 | 主因 top3 |
```

## 步骤 3 — 时间分布 + 集中爆发判断

每个高频错误抓时间分布:
- 散点偶发(每小时个位数,跨全天)— 入故障日志"未修"
- 集中爆发(某 1-2 小时密集) — 先看是不是一次性事件(网络迁机 / 屏保 / 罐头崩),不是事件性才入 bug

## 步骤 4 — 新 bug 入故障日志.txt

仓库 `自动发布/故障日志.txt` "未修"段顶部插条目,格式:
```
--- [时间] [机器] [大件] ---
现象: 一句话
影响: 多少篇/多少账号/多长时间
上下文: 触发条件
推测根因: (可选)
谁记的: 绣虎(Win 台机)
状态: 未修
```

git commit + push,各机 pull 同步。

## 步骤 5 — 已修 bug 标完成

如果 v1101.X patch 已落地解决,把对应条目从"未修"挪到"已修",补:
```
★ 修后:
修法: ...
落地版本: v1101.X
落地时间: ...
落地人: 绣虎 / xxx
```

**Why:** 2026-04-28 缺哥拍 — 之前 bug 散在各机 log 里,没人看就没人修。每日抓一次 + 跨机汇总 + 写故障日志,bug 不丢,修法可追溯,落地度可核对。

**How to apply:**
- 缺哥发起 routine(语句类似"抓 4 mac 昨夜日志" / "看昨晚 bug")→ 直接进步骤 1
- 不要再问"要不要这样" — 流程固定
- 报告短,统计实,不灌水
- 报告完毕 → 等缺哥拍板挑哪个深挖
