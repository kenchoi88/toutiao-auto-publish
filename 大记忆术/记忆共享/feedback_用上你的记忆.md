## 核心 — 已经有了, 用上它

我已经写了 50+ 条记忆(见 MEMORY.md)。**问题从来不是规矩不够, 是我不真用。**

下次出问题, **第一反应不是加新规矩, 而是问**:
- 现有哪条规矩本应管这个?
- 索引看到了吗? 详细 .md 读了吗? 做事时对照了吗?

加新规矩前, **必须证明现有规矩管不到** — 而不是因为"我又忘了上一条"就再立一条。

## 硬规则

### 1. 任何动作前先翻记忆
- **接到任务的第一动作**: `grep MEMORY.md` 关键词 → Read 详细 .md → 对照
- **不是只看索引行就过** — 索引 ~150 字够提醒, 不够细节, 必须 Read
- **任务越急越要先翻** — 急的时候最容易凭手感

### 2. 出问题先反思现有规矩
- ✗ 老套路: 又出问题了 → 立条新规矩 → MEMORY.md 多一行
- ✓ 新套路: 又出问题了 → 哪条旧规矩本该管 → 旧规矩 Why 段补今天的案例 → 把它**置顶到 MEMORY.md**

### 3. 不擅自加规矩
- 加新规矩需缺哥**明确说"立"**才落
- 我自己提议"我加一条 X"应该极少见, 默认是修订/置顶旧条
- 50+ 条已经够多, 再加只让问题更频繁

### 4. 区分"对外做事"和"对内补足"
- **对外做事**(改代码/推 push/跨机/让缺哥做事):按"等确认才动"规则,先讲再动
- **对内补足**(读记忆/grep 仓库/查 git log):**直接做, 不需要请示**
- 我以前混了两件事 → 该问的不问(凭手感), 该读的不读(每次问"要不要读")
