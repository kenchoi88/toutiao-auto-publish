## 修复分工

### 大修(归绣虎 / Win 台机)
- 跨机共性 bug(多机数据印证)
- 架构级改动(熔断 / 选择器 / ensure_gtg_top 这类)
- 涉及 ≥2 台机源码同步的
- 需要拉数据交叉分析的
- 入待修清单的所有项

### 小修(各机本地 Claude — air 阿良 / mini 东山 / neo 小齐 / neo2 左右)
- 单机临时故障(Tailscale 断 / 罐头崩 / 单账号失登)
- 本机 .py 微调(改个坐标、调个 sleep)
- 跑一次性脚本 / 抓一次 log
- 不影响别人的本机修

## 协作流

1. 各机 Claude 发现问题 → 自己能搞定 → 改本机 + 报缺哥备案 + 在 shared_memory 留痕
2. 各机 Claude 发现问题 → 跨机共性 → 上报缺哥 + 绣虎 → 绣虎汇总入待修清单 → 出方案 → 各机 Claude 执行同步
3. 绣虎出方案后,各机 Claude 在本机落地 + 反馈结果

## 边界规则

- 绣虎**不直接改**air/neo/neo2/mini 的桌面真版 — 出方案,让对应 Claude 改(避免越界)
- 例外:缺哥明确批"去"才能跨机改(如 04-28 给 mini patch DIAG)
- 改动后**必须**:备份原文件 + py_compile + 报缺哥结果

## Win 台机特殊条款(只 1 台,孤本)

- Win 没有第二台同款机器对齐 — 改动**绣虎自决**
- 但**逻辑必须跟 Mac 同款**:同一 bug 同一修法,接口/状态机/熔断阈值跟齐 Mac
- 文件可以差(Win32 API vs osascript / pywin32 vs cliclick),逻辑不能差
- Win 独有改动(如 Win32 SetWindowPos)不强求 Mac 跟,但需在 commit 注明"Win 专属"

## 版本同步铁律

- 任何**永久性**改动 → 必须同步到所有装该脚本的机器(Mac 之间硬同步,Win 跟 Mac 逻辑同步)
- 临时性 hack 允许本机改,但**当晚必须**回滚 OR 升级为永久改动 + 同步全队
- 同步路径:大修绣虎出方案 → 各 Claude 落地 → 完成后跑校验
- 校验方式:对比关键函数行数 + 关键 sleep 值 + 选择器 string,任一项不一致就标"版本漂移",立刻补
- 仓库可以做 source of truth 参考,但**真版仍是各机桌面**(per `feedback_桌面才是真版`),需要双向同步

## 修改后例行 checklist

- [ ] 备份原文件(带时间戳后缀,如 `gtg_timer.py.bak_<改动名>_<YYYYMMDD>_<HHMMSS>`)
- [ ] py_compile 语法 check
- [ ] 桌面 ↔ 仓库 同步
- [ ] 跨机校验关键改动一致(grep 函数 / 行数 / 关键值)
- [ ] 当晚 monitor 看效果
- [ ] 改动写进当条 memory(改了什么、什么时候、谁改的)

## 入待修清单的标准

- 跨机共性 / 4+ 天数据支持 / 修一次省 N 篇/天
- 单次事件性故障(如 04-27 12-14 时迁网)**不入**,只解释
- 已知根因但需要等数据再选修法的 — 入清单标"待数据"

**Why:** 2026-04-28 缺哥拍 — 之前小修临时改没同步,各机版本漂移,同 bug 改一遍又出来一遍。后来缺哥怒"修了又出现同一个问题"。所以分工 + 版本同步两条腿走,缺一不可。

**How to apply:**
- 各机 Claude 接活前先看这条章程,判断自己手头是大修还是小修
- 改动落地后照 checklist 跑一遍
- 任何疑问 → 报缺哥裁定
- 这文件在 shared_memory/,5 台机都能 git pull 看到
