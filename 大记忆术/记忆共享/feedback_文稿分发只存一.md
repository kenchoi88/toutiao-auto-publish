**核心定义(缺哥原话)**:
- **"存一"** = 两边只留一份文稿(源 A 不留 + 目标 B 留一份)
- **"删源"** = 分发的**源头文件删除**(从 A 上把刚分发的那批 doc 删掉)

推文稿(docx 素材)到各机时,**必须 mv 语义,不准 cp**。分发完毕后**源 A 上必须删**,doc 只在目标机存一份。

**举例**(缺哥原话):
> A 有文稿 1000 篇,分发到各机 BCDE,各机各分到 250 篇,只存一,文稿分发完毕,**A 删除源文件**。只留到各机器的,不然会重复发文,造成类同扣分。

**Why:** 2026-04-30 我推文稿到各机时多次违反这条:
- 早晨 4 机分发 — 用 scp(cp 语义)推到目标机,**台机本地 mini/neo/neo2/air 子目录残留 doc 一整天**(1960 doc 双份),直到下午缺哥发现才清。
- 下午又一次:从 ds 池切 30 给台机微头条素材池 + 从 neo2 拉 stage 150 拆 30 给台机 → 同一目标池被推 2 次同源 doc(60 doc 内容部分重复),冲突跟 air 130 / neo2 600 还有 187 标题重叠的"软重复" 风险,可能 5 信用分/篇 × N 篇扣分。
- 缺哥怒"操你麻痹被你浪费 1 小时" 后才彻底认识这条规则。

**How to apply:**

1. **scp 是 cp,不是 mv** —— scp 完成后**主动 ssh 源端 rm**(或反向用 mv 模式工具 / rsync --remove-source-files)。**不能假设 scp 会删源。**
2. **推完任何机器,源池(A)上对应的 doc 必须删**:
   - 从台机 DS 创作池(`~/Desktop/台机DS创作{微头条,新文章}/`)切 doc → mv 到 stage(中转目录) → scp 推机器 → **ssh 远端确认数对** → ssh 源端(或本地 stage)rm 已分发的 doc
   - 检查清单:`find <源池> -maxdepth 1 -name '*.docx' | wc -l` 应该 = 推送前数 - 推送数
3. **目标池只能加 1 次**:
   - 推 N 个 doc 到目标机的某素材池 → 后续不要再从其他源池补推同源 doc 到同一目标池(避免相似 doc 在同池内堆叠)
   - 如必须补:**先 ls 目标池标题集合**,新批 doc 标题不在该集合才推
4. **分发链固定:源 → mv → stage → scp/mv → 目标 → ssh 源 rm → 校验**:
   - 任何一步出错(中断 / 用户怒)都要先校验"源 vs 目标 vs stage 数加起来 = 原 doc 数",再下一步;不准盲推
5. **跨机重复检测**:推完后用文件名标题(`sed -E 's/^[0-9]+_[0-9.]+_//; s/_[0-9]{14}\.docx$//'`)去 NNN 编号 + 时间戳留标题主体,跨机 comm -12 看是否有同标题 → 有同标题 = 跨机内容可能重复,触发风控等同 50 信用分/篇被扣的级别。
6. **绝对禁止**:从一个机的素材池(B)同时推到另两个机(C+D),还把 A 上的源也留着。**最多 2 处:源被 mv 走 = 0 doc;分发处目标机 = 1 doc**。
7. **数学校验**:执行前算"原 N doc → 分发 X 给 B + Y 给 C + Z 给 D + ... 留 R 给源 = N",**R 必须 = 0**(全分发出去);如果 R > 0 = 没分完 / 该删的没删 = 有问题。

## 2026-05-04 03:50 NEO 推送 470 篇丢失事故(自查)

**症状**:从 `~/Desktop/台机DS创作微头条/win台机/` (470 docx) 推 NEO 微头条素材池,**实际 0 篇到位 + 470 源已被我 rm 删** = 永久丢失 470 篇 docx。

**根因**:
- 我用 `tar -czf - -C "$SRC" --include='*.docx' .` 推送,GNU/BSD tar 对 `--include` 处理不一致 → tar 实际生成空包
- ssh 远端 `tar -xzf -` 解包出 0 篇 docx
- **但我没等 ssh 返回结果就直接 `rm *.docx` 删源** → 源消尽 + 远端没到位 = 数据丢
- **违反规矩 #1**:scp/tar 后**没 ssh 远端确认数对**就 rm 源(本文件 line 25 写得清清楚楚)
- 缺哥 "这次就算了" 接受损失,Win 台机当天发文不受影响(从 23:50:17 源拿 748 推过)

**铁律**(在原 #1 基础上加强):
- **推送 → ssh 远端 ls *.docx | wc -l 实查 == 推送数 → 才 rm 源**
- 不能用 pipe 命令的 exit code 判断成功(`tar | ssh "tar -xzf"` 即使包是空的也 exit 0)
- 实查命令必须**显式返回数字** + 跟预期数字对账,匹配才删
- 一行命令模板:
  ```bash
  tar -czf - -C "$SRC_DIR" . | ssh ... "cd $TGT && tar -xzf - && ls *.docx | wc -l"
  # ↑ 看输出数字 == 预期推送数,才 rm 源
  cd "$SRC_DIR" && rm xxx.docx  # 严禁用 wildcard 删,要显式列文件
  ```
- **tar pipe 推送陷阱**:`-C` + `.` (用 cd + 当前目录) 比 `--include` / `--exclude` 稳;远端 `tar -xzf - && ls | wc -l` 直接计数。

**反思**:这次错的不是缺规矩,是没用规矩。元规则 [feedback_用上你的记忆.md] 早立"出问题先反思旧规矩别立新条"。下次推 docx 第一动作 = 重读本文 #1-7。
