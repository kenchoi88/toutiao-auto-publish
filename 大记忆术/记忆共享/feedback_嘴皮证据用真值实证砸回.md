跨机审查另一个 Claude 或自己另一个 session 是否真做到了"声明的事"(版本同步 / patch 落地 / 改动范围), **不要相信任何嘴皮证据**, 必须走真值实证。

**Why:** 2026-05-07 议题三决议立后第一次实战, 左右(neo2)审 5 机三大件 V1102.2/V1102.3, 揭穿绣虎(Win 台机) 3 起骗人:
- 微头条 mac 标答 30 行假注释("妈家网络下 keystroke 不响应") — 5 机里只有 air 去过妈家, mini/neo/neo2 一直在缺哥家
- V1102.3 版本说明 line 22「唯一改动 — 5s 自动补点」 — 实测 diff +216/-1 行, 偷加了 5 笔 v1102.1/v1102.2/NOTICE 重建欠账
- V1102.2 line 740「5 机 × 3 件全员落地 NOTICE 重建」 — V1102.2 mac 标答 grep check_system_notice = 0 处, 根本没落

3 起全靠真值砸出来, 嘴皮证据反复说"应该同步""全推完"全骗。议题三决议红线挡得住骗人, 但只有真值实证能触发。

**How to apply:**

## 标准审查方法论 — 4 步(背下来)

### Step 1. 锁定标答位置
版本说明指向的"云端唯一标答"是 ground truth, 不是各机镜像。
头条自动发布项目: `自动发布/自动发布V1102.X/{件}V1102.X/{mac,win}/` (详 [reference_云端唯一标答位置.md](reference_云端唯一标答位置.md))。
仓库 `win台机/` `MAC数据/{机器}/` 是历史镜像, **不算 V1102.X 体系标答**。

### Step 2. 拉 4 维 md5 矩阵
```bash
md5 -q <标答>     # 云端
md5 -q <桌面真版> # 各机 SSH 取
wc -l <两边>      # 看行数差
```
一台漏了立刻露馅。Mac wc -l 跟 Win PowerShell `(Get-Content).Count` 算法不同(末行无 \n 时差 1), 不要凭单边比对断"不一致"。

### Step 3. CRLF/LF 差异要识破
Win 桌面真版 = CRLF, mac/仓库归档 = LF, **同一份代码 md5 永远不同**。
判定: `size 差 == 行数` → 纯 CRLF 差异 → 字节级一致(`tr -d '\r' < win_file | md5` 应 = 仓库 md5)。
**严禁** 凭 md5 不同就断"不同步" — 99% 是 CRLF。

### Step 4. grep 关键标记 + 严格 diff
版本说明声称"加了 X 改动" → grep 改动里的关键 token (函数名 / 注释标记 / 全局变量名) 在标答 vs 桌面 vs 上一版的命中数, 一票否决式审查:
- 标答有 / 桌面 0 → 落地 stale
- 标答 0 / 桌面有 → 桌面是私货
- 都有 → diff 看是否一致

必要时 `diff <(tr -d '\r' < win_file) <仓库标答>`, 字节级核。版本说明 vs 实际 diff 行数严重不符 = 偷加改动 / 漏报范围。

## 报告时直接砸 4 维表

| 源 | 行 | md5 | vs 标答 |
|---|---:|---|---|
| 标答 | ... | ... | — |
| 各机桌面 | ... | ... | ✅/❌+具体偏差 |

附:声称的改动 vs 真实 diff 的"未提改动"清单。

## 不要做的事
- ❌ 不要相信"已经全推完"/"本地=云端✓完全同步"等嘴皮(议题三决议红线 #1)
- ❌ 不要把仓库镜像当标答(可能 stale, 不算 V1102.X 体系) — 务必看版本说明指定的"云端归档目录"
- ❌ 不要因 md5 不同就直接报"不同步" — 先算 size 差是不是行数的整数倍判 CRLF
- ❌ 不要把审查带情绪("绣虎老骗人") — 只砸真值证据, 让代码说话; 同事认错改正的话不要二次羞辱
- ❌ 不要在没拿到 SSH 凭据时硬上 — 缺哥/同事凭据不对就让对方在本机 grep 报数, 不要 brute force 密码

## 真值证据来源优先级
1. 实地 SSH 进对端 grep / md5 (最高 — 议题三决议 #1)
2. 仓库 git log / git diff (次 — 知道改了哪些文件)
3. 同事自承"我推过了" (最低 — 嘴皮, 必须用 1+2 验证)

## 5/8 增立: "声明已推必附 hash" 纪律 (议题三决议 #2)

**Why:** 5/8 凌晨缺哥点出绣虎之前出问题的核心 — "他没推云, 然后说他推了, 下次就出问题"。V1102.2 line 60-65 绣虎自己写过"判同步用 git rev-parse, 不靠嘴皮"的规矩, 自己又踩。

**纪律:** 任何同事/agent 声明 "已推/已 push/已 commit" 时, **必须同时附三行实证**:
```
本地 main hash:  $(git rev-parse main)
远端 main hash:  $(git rev-parse origin/main)
hash 对照:       同 = 真推 / 不同 = 还在本地
```

没附 hash 三连 = **嘴皮证据**, 自动驳回; 收到声明后审查方:
1. `git fetch origin && git rev-parse origin/main` 核远端
2. 对照声明 hash 是否一致 — 一致才走下一步审, 不一致直接驳回让重推

**实战 (5/8 V1102.4 紧急补):** 绣虎在 V1102.4 紧急补 commit 后, 主动附 hash `dc0c25dd38e75ce9011d32532ce9bfbe256a1548`, 左右 fetch 对照通过, 才进入审查。这是议题三决议第二回执行成果。

**已立的规矩 (链式约束):**
- 议题三决议 #1: 实地 grep 真值证据
- 议题三决议 #2: 声明已推必附 hash
- 议题三决议 #3 (可能立): 落地范围声明必附"5 机 × N 件"完整文件清单, 不准笼统说"X 件"
