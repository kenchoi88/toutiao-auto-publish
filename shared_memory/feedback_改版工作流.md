## 缺哥原话

> "先把事件写进故障说明,版本号 v1101.X,修改内容写进版本说明,再写代码,
>  更新本地三大件,验证无误,再同步传仓库,再同步其他 4 台 Mac,
>  再验证所有机器同步版本,以后我们每次改版都这么做。"

## 7 步流程(顺序卡死,不可跳)

### 1. 写故障说明
- 落点:仓库 `故障日志.txt`
- 内容:现象 / 影响 / 上下文 / 真根因 / 跨件确认 / 谁记的 / 状态
- 状态先标"修中(v110X.Y 落地中)",代码落地完再补"已修 + 修法"

### 2. 写版本说明
- 落点:仓库 `版本说明.txt`
- 标版本号(由缺哥定,Claude 不擅自决定 v110X.Y)
- 段位置:插在前一版本之后,「待并入 v1102」之前
- 内容:修的 BUG / 修法 / 落地范围 / 架构红线声明(哪些不动) / 后续观察

### 3. 写代码
- 落点:**改的人所在机的桌面真版**(参见 `feedback_桌面才是真版.md`)
- Win 台机绣虎改 `~/Desktop/台机专用自动发布/<件>/`
- 改前 cp `<file>.bak_pre_v110X.Y_<YYYYMMDD>_<HHMMSS>` 留痕

### 4. 本地验证
- `python -m py_compile <file>` 三件全通
- 条件允许:跑一篇实测(或留待夜跑观察)
- 不通过 → 回滚 .bak,重写

### 5. 同步传仓库
- cp 桌面真版 → 仓库相应子目录(win台机/{微头条,文章,文章定时}自动发布/)
- `git add` + `git commit -m "v110X.Y: <一句话>"` + `git push`
- 仓库的 v110X.Y/ 主线目录(若开)同步落版

### 6. 同步其他 4 台 Mac
- **不要 Win Claude 自己 ssh 进 Mac 改**(违反 `feedback_只动自己文件夹.md`)
- 各 Mac Claude(阿良 air / 小齐+小师弟 neo / 左右 neo2 / 东山 mini)拉仓库后,
  按相同 patch 在自己机器上落地(代码框架略不同,逻辑同款)
- 落地证据:各机 `<file>.bak_pre_v110X.Y_*` + grep patch 标记

### 7. 全机版本验证
- 不靠"应该同步了"嘴皮证据
- `git fetch origin && git rev-parse main origin/main` 两 hash 同 = 仓库已同步
- 各机 ssh 列 `.bak_pre_v110X.Y_*` 时间戳 + grep patch 关键字符串 + md5
- 不要求 md5 完全相同(各机有微调),只验"修法 patch 标记"+ "bak 时间戳"两证据
- 任一机缺则等其上线再补,不擅自跨机覆盖

## Why

- 之前改版常见的 3 种翻车:
  1. 先改代码后忘文档(版本号 / 故障日志 没沉淀,几天后没人记得为什么改)
  2. 文档写了代码没改(空头版本号挂着)
  3. 本地改了仓库没推 / 推了 Mac 没拉(各机版本漂移,memory 「v1101.3 已推全队」就是这种漂移血泪)
- 7 步统一卡死,每步出问题就停,顺序不能跳

## How to apply

- 接到改版指令(显式说"v110X.Y" 或"修这个 bug 出新版")时,**第一句先复述 7 步**给缺哥确认
- 跨步之间报进度,不要一口气做完最后才说
- **第 0 步(隐含):先说明再动**(参见 `feedback_confirm_before_change.md`)
- 第 2 步版本号由缺哥定,Claude 复议不决议
- 任何步出问题(py_compile 不过 / git push 冲突 / Mac 离线 / md5 不一致),**立即停 + 报情况**,不静默修复
