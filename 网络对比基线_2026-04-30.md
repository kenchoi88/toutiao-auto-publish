## 三、关键归因(实证)

### 1. 同机不同网络(mini 04-29 网线 vs 04-30 移动)

| 大件 | 04-29 网线 | 04-30 移动 | 差 | 倍数 |
|---|---:|---:|---:|---:|
| 文章定时 | 73.6s | 76.6s | **+3s** | 1.04x |
| 微头条 | 32.7s | ~55s | **+22s** | 1.7x |

**微头条对网络敏感度 >> 文章定时**(微头条全程跑网络上传/请求,文章定时 30-40s 在本地点击/弹窗,网络占比小)。

### 2. 同网络不同平台(台机 Win/电信线 vs mini Mac/电信线)

文章定时:43.9s vs 73.6s = **差 30s(1.68x)** → Mac 平台单独贡献 30 秒。

### 3. neo2 异常根因(2 重叠加)

| 因素 | 影响 |
|---|---|
| 物理收 Q6 信号弱(同位置 air/neo 满格,只 neo2 不满) | 链路抖动 |
| macOS 自动 fallback 切「客厅电信 WiFi」 | 出口运营商突变 → 头条风控触发 |
| 切回 Q6 满信号后 | 立即降到 81s,跟 mini 同水平 |

## 五、明天(2026-05-01)华为 AX3 Pro 到货后对比方法

### 部署:
1. 华为 AX3 Pro WAN 接**电信光猫**(分一根线出来)
2. LAN 4 口 → 网线 → 4 Mac(每 Mac 接 USB-C 转千兆网卡 if 没自带)
3. 4 Mac **关 WiFi + 删 Q6 / 客厅电信 / 等所有保存的 SSID**(防再次自动 fallback)
4. Q6 留着接移动光猫,WiFi 给手机 / Win 台机用

### 对比维度:

| 验证项 | 期望(04-30 基线 → 05-01 网线后) |
|---|---|
| mini 微头条单篇 | 55s → **~33s**(同 04-29 网线水平) |
| mini 文章定时单篇 | 76.6s → ~73s(只省 3s,网络贡献小) |
| neo 文章定时单篇 | 70s → ~67s |
| neo2 文章定时单篇 | 81s 切回 Q6 / 126s 客厅电信 → **~73-76s 稳定**(消除信号弱 + 自动 fallback) |
| neo2 标准差 | 39.3(电信切换) / 7.4(切回 Q6)→ **< 5**(线稳) |
| 风控告警 | neo2 早晨在客厅电信 WiFi 上跑了 ~150 号 早窗(可能扣分)→ 网线后**单一电信出口,不再触发** |

### 验证步骤(05-01):
1. 接好网线 + 关 4 Mac WiFi → 验证 `networksetup -getMedia en0` 显示 `1000baseT`
2. ping 100.x.x.x 各机连续 30 包,看延迟稳定性(应 < 50ms 全程,不再 800ms 抖动)
3. 跑 1 整轮文章定时(450 篇)→ 对比平均/标准差/失败率
4. 跑 1 批微头条(150 篇)→ 看是否回到 33s 水平
5. 跑 12-14 时常规小旋风历史触发风控时段,看 neo2 是否不再切到电信(因 WiFi 已删)

### 5 个待修缺陷与本次无关(参 故障日志 v1101.x 待修缺陷攒清单):
- 「本轮已发」机制
- 「待补漏」sheet 不 create
- hotfix 跳过 task 不平衡 doc_pool
- 定时下拉 JS 选项点击不可靠
- cliclick 边缘失准

这 5 条等大版本统一改,跟网络改造解耦验证。
